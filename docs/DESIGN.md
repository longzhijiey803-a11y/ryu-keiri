## §A 確定事項（Step 2 以降の不変の前提）

### A-0. 重心
- 作るのは「完成版の経理ソフト」ではなく **「網羅的に拡張できる経理SaaSのUI基盤」**。
- 3本柱：トランザクション・ファースト／エビデンス・ネイティブ／クローズ・ドリブン。
- ポジショニング：会計ソフトではなく **「経理部の業務OS」**。

### A-1. 初期MVP範囲（これだけ。広げない）
1. 共通レイアウト（AppShell / ダークサイドバー / 上部検索バー）
2. ダッシュボード
3. 取引管理（一覧）
4. 証憑アップロード
5. 仕訳候補のUI
6. 承認ステータスのUI
7. 取引詳細の右ドロワー
8. ダミーデータでの一覧・検索・フィルタ・保存ビュー

**明確に後回し（必要時はプレースホルダー）**
本格DB / Supabase Auth / RLS / 銀行API / OCR本実装 / PDF生成 / 法令完全対応 / 固定資産 / 高度レポート / 税務申告連携 / 総勘定元帳・試算表・消費税の専用画面 / 高度ワークフロー / 税理士ポータル / ネイティブモバイル / 多通貨・連結。

### A-2. 「取引」の定義（中核オブジェクト）
- 取引 = 会社のお金・債権・債務・費用・収益に影響する **業務イベント**。
- 例：売上請求 / 受領請求 / 経費申請 / 入金 / 支払 / 振替 / 固定資産取得。
- 取引は次と紐づく **ハブ**：`証憑 / 仕訳 / 承認 / 入出金明細 / 取引先 / コメント / 監査履歴`。
- CRMの営業案件ではなく、**会計・証憑・支払・承認のハブ**として実装する。

### A-4. 法令対応の表現
- 「電帳法・インボイス**完全対応**」とは言わない。**「意識したUI／データ構造」** に留める。
- 扱える構造にするもの：`日付 / 金額 / 取引先 / 証憑種別 / 登録番号 / 操作履歴 / 訂正削除履歴 / 取引・仕訳・証憑の紐づけ`。
- 法令はマーケ文句ではなく **仕様と監査可能性** で語る。

### A-6. Step 2 技術スコープ
- **入れる**：Next.js App Router / TypeScript / Tailwind / shadcn/ui / Lucide / date-fns / RHF / Zod / TanStack Table / Zustand or Context、デザイントークン、共通UI部品、ダミーデータの型、AppShell 土台。
- **入れない**：Supabase / 認証 / DB実装 / RLS / OCR / 銀行API / PDF生成 / 業務ページ中身 / 大量ページ。
- 理由：UIの世界観が固まる前にDBへ入ると画面が技術都合になる。UI/UX最優先。

### A-7. 全Step共通のガードレール（毎回適用）
- その回の範囲外のページ・DB・認証・外部連携は実装しない。必要ならプレースホルダー。
- 品質の低い大量実装より、少数コンポーネントの完成度を優先。
- スコープを勝手に広げない。広げる必要があれば実装前に確認する。

### A-8. UIの一貫契約
- **一覧＝テーブル / カンバン、詳細・作成＝右ドロワー** を全機能で一貫。
- 破壊的・不可逆操作（確定 / ロック / 除却 / 削除）は確認ダイアログ＋監査ログ。
- すべての金額レコードは「証憑ビュー」と「履歴（監査）」を持ちうる、を共通契約とする。

## §B 全体設計（背景・将来拡張の文脈）

### B-1. 想定ユーザー
- 規模：従業員 20〜300名の中小〜中堅企業、経理 1〜10名、外部に顧問税理士。
- ペルソナ：P1 経理担当者（主・毎日）／P2 経理マネージャー・管理者／P3 経営者・役員／P4 一般社員（申請者）／P5 部門長（承認者）／P6 顧問税理士・会計事務所（外部）。
- 設計の軸ユーザーは **P1**。毎日触っても疲れない導線を最優先。

### B-2. 主要業務フロー（将来含む全体像）
- F1 売上〜入金（売掛）：請求発行→売上計上→入金→消込。
- F2 仕入・経費〜支払（買掛）：受領→確認→承認→計上→支払→消込。
- F3 経費精算：申請→一次承認→経理チェック→仕訳→支払。
- F4 仕訳・帳簿：業務イベント→仕訳ドラフト→レビュー→確定→元帳/試算表。
- F5 月次決算（クローズ）：未処理ゼロ確認→残高照合→整理仕訳→レビュー→月次ロック。
- F6 固定資産：取得→台帳→月次償却→除却。
- F7 監査・証憑（横断）：証憑添付→電帳法メタデータ→改ざん検知記録→監査ログ。

### B-3. 画面群（将来の全体像 / 初期は A-1 に限定）
- 共通：ダッシュボード、グローバル検索、通知、My ToDo。
- 取引・仕訳：取引一覧/詳細、仕訳帳、総勘定元帳、試算表。
- 債権：請求書発行、売掛金エイジング、入金消込。
- 債務：受領請求書（カンバン）、買掛金、支払予定、支払消込。
- 入出金：口座残高、明細インポート、マッチング。
- 経費：申請、承認、精算バッチ。
- 承認：承認待ち、承認詳細、ワークフロー定義。
- 決算：月次決算ダッシュボード、整理仕訳、月次ロック。
- 固定資産：台帳、資産詳細、償却実行。
- 税務・帳簿保存：消費税集計、登録番号管理、証憑ライブラリ。
- レポート：財務サマリ、予実、資金繰り、エクスポート。
- 管理：組織/部門、ユーザー/ロール/権限、勘定科目、取引先、税区分、監査ログ、外部連携。

### B-4. UIコンポーネント方針（将来の体系）
- シェル：AppShell / Sidebar / TopBar / GlobalSearch / PageHeader。
- データ表示：DataTable / Kanban / MatchingPanel / LedgerTree / KPICard / TrendChart / StatusTimeline / EmptyState。
- 入力（右ドロワー中心）：Drawer / JournalEditor / InvoiceForm / EvidenceViewer / ApprovalActionBar / Money / AccountPicker / PartnerPicker / TaxClassSelect / DateRange / FileDropzone。
- 横断：StatusBadge / Toast / ConfirmDialog / AuditTrailPanel / ValidationSummary / Notifications。
